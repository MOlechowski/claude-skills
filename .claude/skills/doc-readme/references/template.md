# README Template Reference

This document describes the standard README structure generated by the doc-readme skill.

## Template Structure

```markdown
# {Project Name}

{Description from package file - one line}

## What is {Project Name}?

{2-3 sentence explanation of what this project does and why someone would use it}

## Features

- **Feature 1**: Brief description
- **Feature 2**: Brief description
- **Feature 3**: Brief description

## Installation

<auto>

```bash
{install command}
```

</auto>

## Quick Start

```bash
# Minimal working example
```

## Usage

{Detailed usage examples with code snippets}

## Development

### Prerequisites

- {Runtime and version}

### Setup

```bash
git clone <repository-url>
cd {project-name}
{dev install command}
```

### Testing

<auto>

```bash
{test command}
```

</auto>

## Contributing

{Brief guidelines or link to CONTRIBUTING.md}

## License

{License type}
```

## Section Descriptions

### Required Sections

| Section | Purpose |
|---------|---------|
| Title | Project name as H1 |
| Installation | How to install/add as dependency |
| Usage | How to use the project |

### Recommended Sections

| Section | Purpose |
|---------|---------|
| What is X? | Explain purpose and use case |
| Features | Highlight key capabilities |
| Quick Start | Minimal working example |
| Development | Setup for contributors |
| Contributing | How to contribute |
| License | License information |

## Auto-Generated Content

Content within `<auto>` tags is refreshed when running `update_readme.py`:

- **Installation commands** - Detected from lockfiles (npm/yarn/pnpm/pip/poetry/uv/cargo/go)
- **Test commands** - Based on package manager

Everything outside `<auto>` tags is preserved during updates.

## Customization

After generating a README:

1. Fill in the "What is X?" section
2. List actual features
3. Add real usage examples
4. Customize Quick Start
5. Update prerequisites if needed
